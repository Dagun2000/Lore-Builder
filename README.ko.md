# Lore Builder

*가상 세계의 설정을, 문장 하나씩 일관되게 쌓아갑니다.*

Lore Builder는 가상 세계의 역사와 등장인물을 만들어가는 과정에서 설정 일관성을 지켜주는 도구입니다. 사건이나 사실, 관계를 평문으로 서술하고 관련된 엔티티를 `[대괄호]`로 표시하면, 저장되기 전에 지금까지 기록된 모든 설정과 대조합니다:

```
2100년, [쟝]이 [주점]에서 시비가 붙었다.
```

명백한 모순(이미 죽은 인물의 재등장, 파괴된 물건의 재출현, 장소 자체의 설정과 어긋나는 서술)은 자동으로 반려됩니다. 애매한 경우 — 세계관 규칙 위반 가능성, 인물의 기존 설정과의 충돌, 어떤 상관관계 규칙에 비춰봤을 때 앞뒤가 안 맞는 수치 — 는 이해할 수 있는 근거와 함께 화면에 표시되고, 최종 판단은 사람 몫으로 남습니다. 조용히 저장되거나 조용히 무시되는 일은 없습니다.

엔티티와 그 역사는 나란히 관리됩니다: 구조화된 사실(누가 존재하고, 무슨 일이 있었고, 언제였고, 서로 어떻게 연결되는지)은 SQLite에, 자유 서술은 벡터 저장소에 담아서, 정확한 이름을 몰라도 의미로 관련 설정을 찾을 수 있습니다.

## 왜 필요한가

설정이 어느 정도 쌓이고 나면 머릿속만으로 전부 기억하기가 어려워집니다 — 추방된 인물이 슬그머니 수도로 돌아오거나, 소유자가 죽은 뒤에 벼려진 검이 생기거나, 특정 자격이 있어야 가입 가능한 단체에 조건이 안 맞는 인물이 슬쩍 들어가거나. Lore Builder는 이야기를 대신 써주지 않습니다 — 지금까지 확정한 모든 설정을 기억하고 있다가, 새로 들어오는 내용이 그와 어긋나면 나중에 플롯 홀이 되기 전에 미리 짚어줄 뿐입니다.

스키마, 검증 로직, 프롬프트 어디에도 특정 장르나 세계관을 전제하는 부분은 없습니다 — 여기 실린 샘플 데이터는 우연히 판타지 세계관일 뿐, SF나 현대물, 사극 배경도 동일하게 동작합니다. AI 자신의 표현을 배경에 맞게 바꾸고 싶다면 [커스터마이징](#커스터마이징)을 참고하세요.

## 주요 기능

- **자연어 입력** — 사건이나 사실을 평문으로 서술하고 엔티티를 `[대괄호]`로 표시합니다. 인식되지 않는 태그는 신규 엔티티 생성 절차(카테고리 확인, 필수 필드, 문맥에서 추론 가능한 선택 속성까지)로 안내합니다.
- **위키 페이지가 아니라 타임라인** — 모든 사건, 관계, 되돌릴 수 있는 상태(수감, 저주, 추방 등)는 관련된 모든 엔티티에서 참조되는 하나의 타임라인 항목입니다. "현재 상태" 스냅샷은 어디에도 저장하지 않습니다 — 필요하면 그때그때 타임라인에서 계산합니다. 이 세계관 자체에 고정된 "지금"이라는 개념이 없기 때문입니다 — 역사는 어떤 연도든, 어떤 순서로든 추가할 수 있습니다.
- **결정론적 검증** — 사망/파괴 연도 이후 재등장 같은 생애주기 일관성과 종족-수명 대조를 모델 호출 없이 순수 로직으로 처리합니다.
- **LLM 기반 이해** — 문장을 구조화된 사실(누가, 누구에게, 무엇을, 언제)로 변환하고, 신규 엔티티가 등장하면 생애주기 속성, 지속되는 특성, 그리고 별도로 기록할 가치가 있는 사건을 구분해냅니다.
- **새로 들어오는 모든 내용에 대한 모순 검증** — 세계관 규칙 위반 가능성과 엔티티 자신의 기존 설정과의 충돌을 확인합니다: 명시적 제약, 위험도/성격과 행동 톤의 부조화("목숨이 위험한 곳"에서의 여유로운 피크닉), 심지어 상관관계 규칙에 비춰봤을 때 앞뒤가 안 맞는 수치("많을수록 강하다"는 규칙인데 정작 최하위권이면서 스스로는 특출나다고 주장하는 경우 — 단, 규칙 자체에 명시된 예외는 반영합니다)까지 확인합니다. 이 검증은 연도가 있는 사건뿐 아니라 사건 없이 신규 엔티티만 소개하는 문장에도 동일하게 적용되며, 항상 이유와 함께 승인/반려를 선택할 수 있습니다.
- **변경 하나당 결정 하나** — 하나의 변경은 diff 하나(핵심 사실 + 함께 갱신되는 나머지)로 조립되어, 건드린 행마다 한 번씩이 아니라 승인/반려 결정 한 번으로 처리됩니다.
- **직접 관리하는 목록** — 되돌릴 수 있는 상태와 관계 유형(수감, 추방, 동맹 등)은 코드에 박혀있지 않은 평범한 편집 가능 목록이며, 각 항목에는 이름만으로 AI가 추측할 필요 없도록 실제 의미를 짧게 설명하는 노트가 달려 있습니다. 판단 과정에서 정말로 새로운 유형이 필요해지면, 등록되기 전에 먼저 승인을 요청합니다.
- **창작 모드(Creator Mode)** — 사실 하나가 아니라 이야기 한 토막을 통째로 서술하면, AI가 거기서 나오는 모든 사건을 초안으로 작성하고, 일반 입력이 거치는 것과 동일한 규칙/교차검증으로 스스로의 초안을 검증하며, 통과하지 못하면 다시 시도합니다 — 카테고리별로 허용하면 장면에 필요한 단역 인물이나 사물을 즉석에서 만들어낼 수도 있습니다.
- **엔티티 시각화** — 어떤 엔티티든 지금까지 관여한 모든 사건을 보여주는 타임라인과, 한 다리 건너 연결된 모든 대상을 보여주는 관계도를 데이터에서 바로 생성합니다. 별도로 관리할 것이 없습니다.
- **원하는 LLM을 직접 선택** — OpenAI, Anthropic(Claude), Google(Gemini), 또는 로컬 Ollama 모델 중 환경 변수 하나로 선택할 수 있습니다 — 코드 수정이 필요 없습니다.
- **이중 언어 인터페이스** — GUI 자체의 메뉴와 메시지는 언제든 한국어/영어로 전환할 수 있으며, 이는 세계관 자체가 어떤 언어로 쓰여 있는지와는 완전히 별개입니다.
- **웹 GUI** — 같은 자연어 흐름을 쓰는 채팅 모드와, 모든 엔티티를 훑어보는 딕셔너리 모드를 제공합니다. 엔티티를 열면 전체 상세 정보(필드, 관련 기록, 연결된 대상에 대한 집중 관련성 검색)를 보거나, 타임라인 항목을 그 자리에서 수정하거나, 참조 관계까지 자동으로 정리하며 무언가를 삭제할 수 있습니다.
- **원한다면 CLI로도** — 같은 채팅 흐름과, 기존 엔티티를 직접 수정하는 전용 도구를 제공합니다.
- **나중에 검토할 항목 표시** — 검토 중 나온 내용을 저장 여부와 무관하게 다시 살펴볼 항목으로 표시할 수 있습니다. 해당 엔티티 자체가 실제로 고쳐지면 표시는 자동으로 사라집니다.

## 시작하기

### 요구 사항

- Python 3.12 이상
- 지원되는 LLM 공급자(OpenAI, Anthropic, Google) 중 최소 하나의 API 키 — 또는 API 키가 전혀 필요 없는 로컬 [Ollama](https://ollama.com) 인스턴스. 스키마, 저장소, 결정론적 검증은 어느 쪽이든 LLM 없이도 전부 동작합니다.

### 설치

```bash
python -m venv .venv
source .venv/Scripts/activate    # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 환경 설정

예시 env 파일을 복사한 뒤 필요한 값을 채웁니다:

```bash
cp .env.example .env
```

최소한 `LLM_PROVIDER`(`openai`, `anthropic`, `ollama`, `google` 중 하나)와 해당 공급자의 API 키는 설정해야 합니다 — Ollama라면 API 키 대신 실행 중인 인스턴스를 가리키는 `OLLAMA_BASE_URL`만 있으면 됩니다. `.env.example`의 나머지 값(공급자별 모델 오버라이드, `WORLD_LANGUAGE`)은 기본값만으로도 동작합니다 — `WORLD_LANGUAGE`를 바꾸기 전에는 [커스터마이징](#커스터마이징)을 먼저 읽어보세요.

### 샘플 세계관 데이터 심기

```bash
python scripts/seed_db.py
```

## 사용법

### 웹 GUI

```bash
streamlit run app.py
```

사이드바에는 언어 전환 토글(한국어/English — GUI 자체의 메뉴와 메시지만 바뀌며, 세계관 내용이나 AI가 실제로 쓰는 글에는 영향을 주지 않습니다), 어떤 탭에 있든 항상 동작하는 검색창, 나중에 다시 살펴볼 항목을 모아놓은 **🚩 플래그 확인** 패널, 그리고 두 개의 실제 모드가 있습니다:

- **채팅** — CLI와 동일한 자연어 파이프라인을 프롬프트 대신 위젯으로 보여줍니다. 입력창 위 토글로 **일반 채팅**(사실/사건 하나씩)과 **창작 모드**(이야기 한 토막을 통째로 서술하면 AI가 그에 담긴 모든 사건을 초안으로 작성하고, 스스로 검증하고, 반려되면 다시 시도합니다 — 카테고리별 체크박스로 단역 엔티티를 즉석에서 만들도록 허용할 수도 있습니다)를 전환합니다.
- **딕셔너리** — 모든 카테고리를 훑어봅니다. **Relations/Status** 항목에서는 되돌릴 수 있는 상태/관계 레지스트리를 직접 편집(추가, 설명 수정, 사용 중이 아니면 삭제)할 수 있습니다. 엔티티를 열면 전체 상세 정보(필드, 집중 관련성 검색, 타임라인 참여 내역)를 볼 수 있고, 타임라인 기록이 아닌 엔티티라면 타임라인 스윔레인과 관계도를 보여주는 **시각화** 탭도 함께 제공됩니다.

## 커스터마이징

아래 내용은 전부 설정 파일이나 순수 텍스트 프롬프트이며, 코드 수정이 아닙니다 — 다른 세계관, 다른 장르, 다른 모델 공급자로 바꾸는 일이 로직이 아니라 데이터 편집으로 끝나도록 만드는 것이 이 프로젝트 전체의 설계 의도입니다.

### 엔티티 카테고리 추가/변경하기

`schema_registry.yaml`은 어떤 카테고리가 존재하고 각 카테고리가 어떤 필드를 갖는지에 대한 단일 진실 공급원입니다. 카테고리는 이런 모양입니다:

```yaml
faction:
  id_prefix: faction_
  fields:
    - name: name
      type: text
      required: true
    - name: category
      type: enum
      options: [mercenary_guild, kingdom, religious_order, tribe]
      required: true
    - name: founded_year
      type: integer
      required: false
      role: lifecycle_start
    - name: disbanded_year
      type: integer
      required: false
      role: lifecycle_end
    - name: notes
      type: text
      required: false
    - name: event_ids
      type: list
      required: false
```

필드 타입: `text`, `integer`, `boolean`, `enum`(`options` 목록 필요), `reference`(`ref_category`로 다른 카테고리를 가리키거나, 모든 카테고리를 허용하려면 `any`), `list`. `role: lifecycle_start` / `lifecycle_end` 태그는 해당 필드가 그 카테고리의 "탄생"/"소멸" 연도 역할을 한다는 표시로, 결정론적 검증과 현재 상태 계산에 쓰입니다 — 수명이 있는 새 카테고리라면 이 한 쌍을 그대로 넣어주면 됩니다. 카테고리(또는 기존 카테고리의 필드)를 추가하면 딕셔너리 선택 목록, 엔티티 생성, 태그 매칭, 참조 드롭다운 등 모든 곳에 자동으로 반영됩니다 — 이름을 하드코딩해서 알고 있어야 하는 곳은 없습니다.

### 되돌릴 수 있는 상태와 관계

`status_effects.yaml`은 되돌릴 수 있는 모든 개인 상태(수감, 저주 등)와 관계 유형(동맹, 적대 등)의 레지스트리입니다:

```yaml
- id: imprisoned
  label: 수감
  type: individual
  notes: 물리적으로 수감 장소를 벗어난 행동(다른 지역 방문, 자유로운 이동 등)은 불가능하다. 감방 안에서의 행동(대화, 생각, 식사 등)은 가능하다.
```

`type`은 대상이 없는 상태(예: *저주받음*)라면 `individual`, 다른 엔티티를 대상으로 하는 관계(예: *~에서 추방됨*, *~와 동맹*)라면 `relational`입니다. `notes`는 그 상태가 실제로 무엇을 의미하는지 서술하는 자유 텍스트로, 관련된 모든 LLM 검증에 그대로 전달됩니다 — 활성화된 동안 무엇이 가능하고 무엇이 불가능한지 구체적으로 적을수록 검증 품질이 좋아집니다. 파일을 직접 편집하거나, GUI의 **딕셔너리 → Relations/Status**에서 항목 추가/설명 수정/삭제(사용 중이면 경고와 함께 차단)를 할 수 있습니다. 대화 도중 자연스럽게 등장한 정말로 새로운 관계 유형은 레지스트리에 영구히 추가되기 전에 먼저 승인을 요청하므로, 모르는 사이에 새 항목이 생기는 일은 없습니다.

### 시드 세계관 데이터 편집하기

`db/{category}/*.md`는 `scripts/seed_db.py`를 새로 실행할 때마다 불러오는 시작 데이터입니다. 각 파일은 YAML 프론트매터 블록(`id`와 해당 카테고리가 정의하는 필드들) 뒤에 관련성 검색에 쓰이는 자유 서술 본문이 이어지는 구조입니다:

```
---
id: char_쟝
name: 쟝
...
event_ids:
  - event_쟝_2080
---

쟝(Jang)은 용병 길드 소속의 인간 용병이다. ...
```

직접 편집할 때 유의할 점:

- **참조는 수동이며 양방향입니다.** 어떤 캐릭터를 참조하는 타임라인 사건을 추가했다면, 그 사건의 id를 해당 캐릭터의 `event_ids` 목록에도(그리고 그 사건에 관여하는 다른 모든 엔티티에도) 추가해야 합니다 — 앱을 통해 저장할 때와 달리, 직접 편집한 파일은 아무것도 자동으로 동기화되지 않습니다.
- **id는 엔티티 이름과 같은 표기 체계를 쓰세요.** 샘플 데이터의 id는 엔티티 이름을 한글 그대로 슬러그화한 것입니다(`char_jang`이 아니라 `char_쟝`) — 파이프라인을 통해 실제로 생성되는 엔티티와 동일한 방식입니다. 표기 체계를 섞으면(이름은 한글인데 id는 로마자인 경우) 정확히 예전에 AI가 실제 이름 대신 id에서 추측한, 뭉개진 한글 표기를 만들어내는 원인이 된 적이 있습니다 — 새 id는 `name` 필드와 같은 표기 체계를 유지하세요.
- **편집 후에는 다시 심으세요.** `db/lore.db`와 `chroma_store/` 디렉터리를 삭제한 뒤 `python scripts/seed_db.py`를 다시 실행합니다 — 시딩은 삭제한 파일에 대응하는 기존 행을 자동으로 정리해주지 않습니다.

### LLM 공급자 선택하기

`.env`의 `LLM_PROVIDER`를 `openai`, `anthropic`, `ollama`, `google` 중 하나로 설정하세요. `.env.example`에 각 공급자의 API 키 변수와, 티어별(저렴한 "simple" 모델과 더 강력한 "reasoning" 모델) 선택적 모델 오버라이드가 모두 정리되어 있습니다. Ollama는 API 키가 필요 없고, 실행 중인 인스턴스를 가리키는 `OLLAMA_BASE_URL`만 있으면 됩니다.

`PARALLEL_RAG_CHECKS`(기본값 `true`)는 파이프라인의 독립적인 두 검증을 동시에 실행할지를 결정합니다 — 클라우드 공급자에서는 실질적인 지연 시간 단축 효과가 있지만, VRAM을 꽉 채우는 크기의 로컬 Ollama 모델을 쓰고 있다면 `false`로 설정하세요 — 두 개의 생성이 동시에 실행되면 실제로 병렬화되는 대신 같은 GPU 메모리를 두고 경합할 수 있습니다.

### 세계관 언어 설정하기

`.env`의 `WORLD_LANGUAGE`(기본값 `ko`, 샘플 세계관과 일치)는 여러분의 세계관 내용 — 그리고 `schema_registry.yaml`/`status_effects.yaml`에 직접 적어 넣는 enum 옵션들 — 이 어떤 언어로 되어 있는지를 선언하는 값입니다. **데이터를 입력하기 전에 딱 한 번만 설정하세요.** 프로젝트 도중에 바꾸면 이미 저장된 값이 그 사이 바뀐 옵션 목록과 어긋나거나, 하나의 세계관 안에 두 언어가 섞여 들어갈 위험이 있습니다. 현재 이 값은 선언만 되어 있을 뿐 아직 강제되지는 않습니다 — AI 자신의 프롬프트는 이 값과 무관하게 여전히 한국어로 고정되어 있습니다. 직접 바꾸고 싶다면 다음 항목을 참고하세요.

### 인터페이스 언어

사이드바의 한국어/English 토글은 GUI 자체의 라벨, 버튼, 메시지에만 영향을 줍니다 — `WORLD_LANGUAGE`와는 완전히 독립적이며, 여러분이 입력하는 내용이나 저장되는 값, AI가 실제로 작성하는 글에는 전혀 관여하지 않습니다.

### AI 프롬프트를 여러분의 세계관에 맞게 바꾸기

스키마와 검증 로직은 장르를 전제하지 않지만, LLM 프롬프트는 짧은 역할 설명으로 시작하며 현재 그 문구가 "판타지 세계관"으로 고정되어 있습니다 — `src/creator.py`, `src/field_update.py`, `src/inference.py`, `src/mapping.py`, `src/rag_check.py`에서 이 문구를 검색해서, AI의 표현이 여러분의 배경(SF, 현대물, 사극 등 무엇이든)과 맞기를 원한다면 직접 원하는 대로 바꾸면 됩니다. 프롬프트, 스키마, 검증 로직의 다른 어떤 부분도 장르가 바뀐다고 해서 함께 바꿀 필요는 없습니다.

## 프로젝트 구조

```
schema_registry.yaml       # 엔티티 카테고리, 필드, 타입 정의 — 단일 진실 공급원
status_effects.yaml        # 되돌릴 수 있는 상태와 관계 유형 (수감, 추방, ...)
.env.example                 # LLM 공급자 + 세계관 언어 설정 템플릿
db/                         # 시드 데이터(YAML 프론트매터가 있는 마크다운) + 생성되는 SQLite/Chroma 저장소
scripts/seed_db.py          # db/*.md를 SQLite + Chroma에 적재
app.py                      # Streamlit GUI 진입점

src/
  config.py                  # LLM_PROVIDER -> 실제 채팅 모델 팩토리 (OpenAI/Anthropic/Ollama/Google)
  settings.py                  # WORLD_LANGUAGE 리더
  i18n.py                        # 인터페이스 언어(한국어/English) 번역 레지스트리
  schema.py                       # schema_registry.yaml / status_effects.yaml 로더 + 조회 헬퍼
  storage.py                       # SQLite + Chroma 저장소 계층
  hard_check.py                     # 결정론적 생애주기/수명 검증
  parser.py                          # 정규식 기반 입력 파싱
  mapping.py                          # 엔티티 태그 매칭 + 신규 엔티티 생성
  inference.py                         # LLM 사건/속성 추론
  rag_check.py                          # LLM 기반 세계관 규칙/설정/상태 교차검증
  archivist.py                           # 추론+검증 결과를 검토 가능한 diff로 조립
  pipeline_session.py                     # 파이프라인 상태 머신 (GUI와 CLI가 공유)
  creator.py                               # 창작 모드: 다중 사건 서사 초안 작성 + 검증
  creator_session.py                        # 창작 모드 전용 일시정지/재개 상태 머신
  visualization.py                           # GUI용 타임라인/관계도 데이터 준비
  deletion.py                                 # 포인터 정리를 포함한 엔티티/이벤트 삭제
  main.py                                      # CLI 진입점
  approval.py                                  # CLI 승인/검토 프롬프트
  field_update.py                               # 기존 엔티티 필드 수정 로직
  detail_panel.py                                # CLI 엔티티 수정 진입점
  flags.py                                        # "나중에 검토" 기록 관리

tests/                         # pytest 스위트 (다수의 테스트가 LLM 공급자를 호출합니다)
```

## 테스트

```bash
pytest
```

LLM을 사용하는 레이어를 검증하는 테스트는 설정된 `LLM_PROVIDER`를 대상으로 실제 API 호출을 발생시킵니다 — 기본적으로 전체 스위트가 아니라 관련된 파일/테스트만 실행하는 것을 권장합니다. 결정론적 레이어(스키마, 저장소, 하드체크, diff 조립, 플래그)는 공급자 설정 없이도 전부 통과합니다.
