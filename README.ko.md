# Lore Builder

*판타지 세계관 설정 일관성 검증 파이프라인*

Lore Builder는 세계관에서 방금 벌어진 일을 한 줄로 입력받습니다 — *"2100년, [쟝]이 [주점]에서 시비가 붙었다"* — 그리고 그 사건을 저장소에 반영하기 전에 지금까지 쌓인 모든 설정과 모순되지 않는지 확인합니다. 명백한 모순(이미 죽은 인물의 재등장, 파괴된 아이템의 재출현)은 확인 절차 없이 즉시 반려됩니다. 애매한 경우 — 세계관 규칙 위반 가능성, 인물의 기존 설정과의 충돌, 심지어 "서클이 많을수록 강하다"는 상관관계 규칙에 비해 수치가 최소치권인데 스스로 "엄청나게 강력하다"고 주장하는 것 같은 수치-서술 불일치까지 — 는 사람이 최종 판단할 수 있도록 화면에 보여줄 뿐, 자동으로 반영되는 일은 없습니다.

두 종류의 지식을 동시에 관리합니다: SQLite에는 구조화된 사실(엔티티들과, 그 엔티티들을 잇는 사건·관계·상태의 타임라인)을, Chroma 벡터 저장소에는 자유 서술 텍스트를 담아서, 정확한 ID를 몰라도 의미로 관련 설정을 찾을 수 있게 합니다.

별도의 "관계" 테이블은 없습니다 — 한 시점에 벌어진 사건이든, 계속 이어지는 관계나 되돌릴 수 있는 개인 상태든 모두 `timeline` 카테고리의 레코드 하나로 표현되고, 관련된 각 엔티티에서 참조됩니다. 엔티티는 "현재 상태" 스냅샷을 저장하지 않습니다(`current_owner`도, `current_status`도 없습니다) — 그런 값이 필요하면 타임라인에서 그때그때 계산합니다. 이 세계관 자체에 고정된 "지금"이라는 개념이 없기 때문입니다 — 연도는 어떤 순서로든, 어떤 범위로든 입력될 수 있습니다.

## 주요 기능

- **자연어 사건 입력** — `[대괄호]`로 감싼 태그를 포함한 평문으로 사건을 입력하면, 파이프라인이 각 태그를 기존 엔티티에 매칭하거나 신규 생성 절차(카테고리 확인, 필수 필드 입력, 문맥에서 자동 추출 가능한 선택 속성까지)로 안내합니다.
- **사건 중심 타임라인** — `timeline` 카테고리 하나가 한 시점의 사건(싸움, 창단 등)과 계속 이어지는 기간형 레코드(관계, 수감/저주 같은 되돌릴 수 있는 개인 상태)를 모두 담습니다. 각각은 entity/predicate/target/start/end로 구성된 하나의 튜플일 뿐입니다. 한 문장에 여러 사실이 응집된 하나의 장면이면 여러 개의 연결된 레코드가 한 번에 만들어지고, 정말로 서로 무관한 여러 사건이 섞여 있으면 추측 대신 사용자에게 나눠서 다시 입력하도록 안내합니다.
- **결정론적 하드체크** — 생몰년/파괴·해체 연도 이후 재등장 같은 시간축 일관성 검증과 종족 수명 대조 검증을 LLM 없이 100% 규칙 기반으로 수행합니다.
- **LLM 기반 추론** — 문장을 구조화된 타임라인 레코드로 변환하고, 신규 엔티티가 등장하면 생몰/창단/제작 연도 같은 생애주기 속성, 지속되는 특성(비고), 남은 시간축 사건을 분리해냅니다 — 전부 이미 존재하거나 지금 막 만들어지는 엔티티에만 엄격히 근거합니다.
- **모든 새로운 주장에 대한 RAG 교차검증** — 세계관 규칙 위반 가능성과 엔티티의 기존 설정(저장된 필드 포함)과의 모순을 판단합니다. 명시적 제약(출입 제한, 능력 요건) 위반뿐 아니라 위험도/성격과 행동 톤의 부조화("목숨이 위험한 곳"에서의 여유로운 피크닉), 그리고 상관관계 자기모순("많을수록 강하다"는 규칙인데 정작 수치는 하위권이면서 스스로 "엄청나게 강력하다"고 주장하는 경우 — 단, 규칙 자체에 명시된 예외 조항은 반영합니다)까지 확인합니다. 이 검증은 연도가 있는 사건뿐 아니라, 사건 없이 신규 엔티티만 소개하는 문장에도 똑같이 적용됩니다 — 신규 엔티티의 주장도 사건과 마찬가지로 신뢰받기 전에 검증을 거칩니다. 모든 판단에는 사람이 읽을 수 있는 근거와 승인/반려 프롬프트가 함께 제공되며, 조용히 반영되거나 조용히 무시되는 일은 없습니다.
- **저장 전 전체 검토** — 하나의 변경은 diff 하나(핵심 레코드 + 함께 갱신되는 엔티티들)로 조립되어, 건드린 행마다 한 번씩이 아니라 승인/반려 결정 한 번으로 저장소에 반영됩니다.
- **웹 GUI (Streamlit)** — 같은 자연어 파이프라인을 쓰는 채팅 모드와, 카테고리별로 모든 엔티티를 훑어보는 딕셔너리 모드를 제공합니다. 엔티티를 열면 현재 필드 값, 하드체크 검토와 연결된 엔티티에 대한 1-hop 관련성 검색이 포함된 필드 수정 화면, 타임라인 레코드는 그 자리에서 바로 수정, 엔티티/이벤트 삭제(포인터 자동 정리 포함)까지 가능합니다.
- **CLI 대안** — GUI 대신 쓰고 싶다면 같은 채팅 파이프라인을 제공하는 `src/main.py`와, 기존 엔티티 필드 수정용 `src/detail_panel.py`가 있습니다.
- **나중에 검토할 항목 표시** — 검토 중 나온 관련 기록을 저장 여부와 무관하게 플래그로 남길 수 있습니다. 플래그가 걸린 엔티티 자신을 고쳐서 저장하면 그 플래그는 자동으로 해제됩니다.

## 요구 사항

- Python 3.12 이상
- OpenAI API 키 (관계/사건 추론과 RAG 교차검증 단계에서만 필요합니다 — 스키마, 저장소, 하드체크는 키 없이도 전부 동작합니다)

## 설치

```bash
python -m venv .venv
source .venv/Scripts/activate    # Windows: .venv\Scripts\activate
pip install langchain-openai chromadb streamlit pyyaml python-dotenv pytest
```

## 환경 설정

프로젝트 루트에 `.env` 파일을 만듭니다:

```
OPENAI_API_KEY=sk-...
```

## 사용법

### 웹 GUI

```bash
streamlit run app.py
```

사이드바 상단에는 어떤 모드에 있든 항상 동작하는 검색창이 있고, 그 아래 두 개의 실제 모드가 있습니다: **채팅** — CLI와 동일한 자연어 사건 파이프라인을 프롬프트 대신 위젯으로 보여줍니다. **딕셔너리** — 모든 카테고리를 훑어보고, 엔티티를 열어 상세 화면(현재 필드 값, 필드 수정, 항목별 플래그 체크박스가 있는 관련 기록 검색, 삭제)을 보거나, 타임라인 레코드를 열어 그 자리에서 수정하거나 삭제할 수 있습니다. 세 번째 탭(시각화)은 아직 자리만 마련된 상태입니다.

### CLI: 새 사건 기록하기

```bash
python src/main.py
```

```
Lore Builder CLI. 종료하려면 '종료'를 입력하세요.

입력> 2100년, [미라]가 [은빛도시]에서 산책을 했다.
CREATE timeline: event_2100년_미라가_은빛도시에서_산책했다
  근거: 새 사건 기록: 2100년 미라가 은빛도시에서 산책했다.
  필드: {'year': 2100, 'location': 'loc_silver_city', 'notes': '2100년 미라가 은빛도시에서 산책했다.'}
  함께 갱신되는 엔티티: char_mira, loc_silver_city
저장하시겠습니까? [저장/취소]: 저장
저장 완료: event_2100년_미라가_은빛도시에서_산책했다, char_mira(갱신), loc_silver_city(갱신)
```

새로운 엔티티 태그가 있으면 카테고리 확인, 필수 필드 입력을 먼저 거친 뒤 같은 검토 단계로 이어집니다. 종료하려면 `종료`를 입력하면 됩니다.

### CLI: 기존 엔티티 수정하기

```bash
python src/detail_panel.py
```

이름이나 id로 엔티티를 찾고, 필드를 선택하고, 새 값을 입력합니다 — 이 엔티티가 걸려있는 모든 사건이 저장 전에 나열됩니다 (GUI의 필드 수정 화면은 대신 관련성 판단을 거친 1-hop 검색 결과만 좁혀서 보여줍니다). 아무 프롬프트에서나 `목록`을 입력하면 지금까지 표시해둔 플래그 목록을 볼 수 있습니다.

### 샘플 세계관 데이터 심기

```bash
python scripts/seed_db.py
```

## 프로젝트 구조

```
schema_registry.yaml       # 엔티티 카테고리, 필드, 타입 정의 — 단일 진실 공급원
status_effects.yaml        # 되돌릴 수 있는 상태의 고정 목록 (수감, 저주, 분실, ...)
db/                         # 시드 데이터(YAML 프론트매터가 있는 마크다운) + 생성되는 SQLite/Chroma 저장소
scripts/seed_db.py          # db/*.md를 SQLite + Chroma에 적재
app.py                      # Streamlit GUI 진입점

src/
  config.py                  # 모델 티어 -> 실제 모델명 매핑
  schema.py                   # schema_registry.yaml 로더 + 조회 헬퍼
  storage.py                   # SQLite + Chroma 저장소 계층
  hard_check.py                 # 결정론적 시간축/수명 검증
  parser.py                      # 정규식 기반 입력 파싱
  mapping.py                      # 엔티티 태그 매칭 + 신규 엔티티 생성
  inference.py                     # LLM 사건/속성 추론
  rag_check.py                      # LLM 기반 세계관 규칙/설정/상태 교차검증
  archivist.py                       # 추론+검증 결과를 검토 가능한 diff로 조립
  approval.py                         # 레거시 블로킹 승인 루프 (main.run_pipeline에서 아직 사용)
  pipeline_session.py                  # 제너레이터 기반 파이프라인 상태 머신 (GUI와 CLI 채팅 루프가 공유)
  deletion.py                           # 포인터 정리를 포함한 엔티티/이벤트 삭제
  main.py                                # CLI 진입점 (채팅 루프 + 사건 파이프라인)
  field_update.py                         # 기존 엔티티 필드 수정 로직 (하드체크 재실행 + 1-hop 관련성 검색)
  detail_panel.py                          # CLI 엔티티 수정 진입점
  flags.py                                  # "나중에 검토" 기록 관리

tests/                         # pytest 스위트 (다수의 테스트가 OpenAI API를 호출합니다)
```

## 테스트

```bash
pytest
```

LLM을 사용하는 레이어를 검증하는 테스트는 `OPENAI_API_KEY`가 필요하고 실제 API 호출이 발생합니다 — 기본적으로 전체 스위트가 아니라 관련된 파일/테스트만 targeted로 실행하는 것을 권장합니다. 결정론적 레이어(스키마, 저장소, 하드체크, diff 조립, 플래그)는 키 없이도 전부 통과합니다.
